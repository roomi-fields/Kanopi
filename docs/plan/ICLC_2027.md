## Ce qu'on soumet

### 1. Long paper (5-12 pages)

**Titre provisoire** : "Kanopi: A Grammar-Based Meta-Sequencer for Multi-Runtime Live Coding"

#### Pourquoi c'est nouveau a ICLC

Aucun paper ICLC dans l'histoire de la conference n'a presente un systeme de live coding
base sur les grammaires formelles (context-free, rewriting). Zero precedent dans les proceedings.

- Alex McLean (createur de Tidal) utilise des patterns algebriques, pas des grammaires generatives
- Sardine identifie le probleme de sync multi-runtime mais ne le resout pas
- L'inference de structure (analyse inverse) est un sujet de recherche ouvert
- Les 3 types temporels (gate/trigger/cv) n'ont pas d'equivalent dans les outils de live coding

#### Structure du paper

**1. Introduction** — le probleme : les sequenceurs sont plats, le live coding
est mono-runtime, les structures temporelles complexes (polymetrie, grammaires
conditionnelles) sont inaccessibles en performance

**2. Background et travaux connexes**
- Bernard Bel et le Bol Processor (40 ans de grammaires formelles pour la musique)
- Live coding : TidalCycles, Strudel, Sonic Pi, SuperCollider, Sardine
- Grammaires formelles en musique : PCFG, grammaires de Lindenmayer
- Gap : personne n'a connecte les deux (grammaires generatives + live coding)

**3. BPscript : le langage**
- 3 mots reserves (gate, trigger, cv), 24 symboles, 7 operateurs
- Double declaration (type temporel + runtime binding)
- Polymetrie comme primitive du langage `{ voix1, voix2 }`
- Backticks : code natif dans le flux temporel
- 5 couches de pitch (alphabet → octaves → temperament → tuning → resolver)
- Cross-culturel : raga, maqam, gamelan, western dans le meme espace

**4. Kanopi : le framework**
- Architecture 3 niveaux (web pur / web enrichi / bridge local)
- Pipeline : source → parser → AST → BPx engine → DerivationTree → dispatcher → transports
- Multi-runtime : SC, Tidal, Python, MIDI, WebAudio, DMX dans un seul fichier
- @map : CC/OSC ↔ triggers/flags (communication bidirectionnelle)
- @scene : hierarchie de scenes avec scoping des flags
- Timeline Canvas : visualisation + deformation temporelle interactive

**5. BPx : le moteur de derivation**
- Instance isolee, AST direct, DerivationTree structure
- 7 modes de derivation (ORD, RND, LIN, SUB, SUB1, TEM, POSLONG)
- Polymetrie iterative (pas de stack overflow)
- FlagStore observable, TriggerBus async
- Live coding : addRule, modifyRule, hotSwap en temps reel

**6. Exemples et performances**
- Scene BPscript orchestrant 3+ runtimes en live
- Comparaison avec TidalCycles/SuperCollider sur les memes structures
- Demonstration de la polymetrie impossible dans les autres outils
- Cross-culturel : raga + tabla + lumieres DMX dans un seul fichier

**7. Discussion**
- Le changement de paradigme : sequenceurs plats → sequenceurs structurels
- Heritage de Bernard Bel et lien avec l'ethnomusicologie formelle
- Limites actuelles et travaux futurs
- Inference de structure (import MIDI → grammaire)
- Vision : le DAW des live coders

**8. References**

#### Pertinence par rapport au theme "Dialogues"

Le theme 2027 est "Dialogues" — interactions reciproques entre codeurs,
machines et audiences. BPscript/Kanopi est litteralement un systeme de dialogues :

- **Dialogue entre langages** : backticks connectent SC, Tidal, Python dans un seul flux
- **Dialogue entre traditions** : raga et western dans le meme espace formel
- **Dialogue humain-machine** : les grammaires generent, le performer guide via flags
- **Dialogue entre scenes** : triggers et flags traversent la hierarchie
- **Dialogue avec le public** : le code projete + la timeline rendent la structure visible

Les topics 2027 mentionnent explicitement :
- "Multiple coder ensemble exchanges" → nos @scene + triggers inter-scenes
- "Lighting systems and DMX control" → notre transport DMX
- "Robotics and physical device orchestration" → nos sorties CV/gate eurorack
- "Classical musician collaborations using live-coded scores" → BPscript comme partition generative

### 2. Performance live (800 mots + rider technique)

**Titre** : "Fields — a grammar-derived live performance"

**Description** : performance solo orchestrant SuperCollider (melodie generative),
Tidal/Strudel (rythmes), WebAudio (synthese), MIDI hardware (Osmose ou controleur),
et visuels (Hydra) — le tout pilote par un seul fichier BPscript.

**Ce que le public voit** :
- Ecran projete : le code BPscript + la timeline Canvas
- Modification live des regles → la structure change en temps reel
- Les flags changent via controleur physique → les guards re-evaluent
- La polymetrie se deforme visuellement sur la timeline

**Rider technique** :
- Laptop + interface audio + ecran de scene + projecteur
- Controleur MIDI/OSC (Electra One MK2 ou equiv.)
- Osmose (clavier MPE expressif)
- Sortie stereo + possibilite quadriphonie

### 3. Demo/poster (complementaire)

**Titre** : "BPscript: 3 Words, 24 Symbols, Infinite Structures"

Demonstration interactive : les participants ecrivent une scene BPscript
dans le navigateur et l'entendent immediatement. Progression de
`S -> Sa Re Ga Pa` a une polymetrie multi-runtime en 5 minutes.
